## 1. 동기 — 쉘판에 남은 잔여 리스크

v1.2.1 쉘 launcher는 가시성·경합·위장은 해소했으나, **쉘로는 이식적으로 못 푸는** 잔여 리스크가 있다:

| ID | 잔여 리스크 | 쉘 한계 | Node 해결 |
|----|------------|---------|----------|
| **N2** | `timeout`/`gtimeout` 부재 + 리뷰어 hang → `wait` 무한 블로킹 | macOS 순정에 GNU timeout 없음. 자체 `sleep&kill` 워치독은 오탐 kill 위험으로 미채택 | `AbortController`+`setTimeout`+`child.kill()` = 외부 의존 0, 오탐 없는 정밀 타임아웃 |
| **N6** | `pkill -P $$` 직속자식만 → 리뷰어 손자 프로세스 고아 | 쉘에서 프로세스그룹 정리가 비이식적·위험(`kill -- -$$`는 부모 동반사망) | `spawn(detached:true)` + `process.kill(-child.pid)` = 자식 그룹만 정밀 정리 |
| **N7** | status JSON 문자열 수기조립 → escaping 취약 | 쉘엔 JSON 직렬화기 없음 | `JSON.stringify` = escaping 무료·구조보장 |
| **테스트** | mock PATH e2e(간접) | 실모듈 단위테스트 불가 | 함수 단위테스트 + 실CLI smoke |

**N2가 핵심.** 현재 쉘판은 N2를 *오케스트레이터 fallback wakeup*(defense-in-depth)으로만 방어 — launcher 자체는 여전히 무방비. Node는 launcher 내부에서 근본 차단.

## 2. 왜 TypeScript 아니라 `.mjs`인가

- **TS = 트랜스파일 단계 필요**(tsconfig/tsx/빌드). 이 팩토리는 **생성 템플릿** — 타겟 프로젝트에 스킬을 찍어낼 때 빌드 스텝을 끼우면 슬림·zero-toolchain 원칙 위반.
- 타입 이득 < 빌드 비용. **순수 ESM `.mjs`(무빌드·무의존)** 가 실익(N2/N6/N7/테스트)을 90% 회수하면서 빌드는 0.
- 따라서 올바른 비교는 "쉘 vs TS"가 아니라 **"인라인 쉘 vs 번들 `.mjs`"**.

## 3. node 런타임 가용성 — 전제 정정(자체검토)

⚠️ **실측 정정:** 초안의 "리뷰어가 전부 node라 node 보장" 전제는 **부분 거짓**. 실측 결과:
- `codex`·`gemini` = node 스크립트(`#!/usr/bin/env node`).
- `claude`·`agy` = **네이티브 Mach-O 바이너리**(node 불요).

따라서 node 보장은 *조건부*다:
- **Codex 런타임**(러너=codex=node) → node 항상 present. ✓
- **Claude Code 런타임**(러너=claude=네이티브) → 일반 리뷰어가 codex(node)면 present ✓. 그러나 **codex 부재로 REVIEWERS=agy 단독**이면 node 미보장(둘 다 네이티브). ✗ 엣지 존재.

→ node 부재 가능 경로가 실재 → **§6 쉘 폴백은 선택이 아니라 필수**. 이 정정이 `.mjs` 단독 전환의 핵심 약점이며 외부감사 핵심 쟁점(§9-2).

## 4. 제안 인터페이스

`skills/external-review-loop/scripts/launch-review.mjs` (스킬 번들, 로딩 없이 실행 = 컨텍스트 무게 0):

```
node scripts/launch-review.mjs \
  --stage {단계ID} \
  --reviewers "codex agy" \      # check-review-tools.sh REVIEWERS: 줄 그대로
  --dir _workspace/reviews \
  --timeout 600                  # 초. 0이면 무제한(비권장)
```

오케스트레이터는 이 명령을 **`Bash(run_in_background: true)`로 launch**(쉘판과 동일한 가시성 모델 — launch/await/poll·fallback wakeup 그대로). 차이는 launcher 내부 구현뿐.

**동작:**
1. `running` status 즉시 쓰기(atomic) — `{status,reviewers,started,results:{}}`.
2. REVIEWERS의 리뷰어별 자식 spawn(`detached:true`, stdin `/dev/null`, stdout/stderr → `_{tool}.md`). 일반 리뷰어 codex XOR claude(러너 제외), 성능 agy(or gemini legacy).
3. 각 자식에 `setTimeout(timeout*1000)` → 만료 시 `process.kill(-child.pid,'SIGTERM')`(그룹), 유예 후 SIGKILL. rc=124(타임아웃 관례)로 기록.
4. 전 자식 settle 후 rc 취합 → `completed|partial|failed`(미실행=failed, 위장금지). atomic 최종 status 쓰기.
5. stdout에 `DONE: status=... ok=.. fail=..`(launch 모드서 tool result로 회수).

**status 스키마는 쉘판과 동일**(Step 3 파서 불변) — 구현만 교체, 계약 유지.

## 8. 결정 기준

**채택 조건(AND):** §7-1 패리티 100% + §7-2 N2 실증 통과 + 외부감사 2R 수렴(`converged-good`) + 슬림 위반 없음(무빌드·무의존 유지).
**기각 조건(OR):** node 부재 환경이 유의미하게 존재 판명 / `.mjs`가 쉘 대비 새 결함 순증 / 빌드·의존 끌어들임.

## 8-1. 자체검토 발견 약점 (외부감사 전 선반영)

1. **node 전제 부분거짓** — §3 정정. claude/agy 네이티브 → node 부재 경로 실재 → 쉘 폴백 필수.
2. **detached 자식의 역(逆)고아** — `detached:true` 자식은 **launcher가 죽어도 살아남는다**(쉘 `trap pkill`이 잡던 launcher-사망 정리가 사라짐). launcher가 `SIGTERM`/`SIGINT`/`exit` 핸들러로 자기 자식 그룹을 kill해야 함(오케스트레이터 abort 시 좀비 방지). 설계에 핸들러 명시 필수.
3. **리뷰어 커맨드 테이블 그대로 운반** — 쉘판의 도구별 정확한 플래그(`codex exec --sandbox read-only` / `claude -p --permission-mode plan --allowedTools …` / `agy -p --model "Gemini 3.1 Pro (High)" --sandbox --print-timeout`)를 `.mjs`가 **동일하게** 인코딩해야 함(추상화하면 안 됨 — 플래그가 격리·권한의 핵심).
4. **이중 보유 비용** — 쉘 폴백 필수(§3) ⇒ 쉘+`.mjs` 둘 다 유지 = 감사면적·유지보수 2배. 이게 N2 근본해결의 값어치를 상쇄하는지가 채택 핵심 트레이드오프(§9-4).

## 9. 오픈 퀘스천 (외부감사 대상)

1. `spawn(detached:true)` + `process.kill(-pid)`가 macOS·Linux 양쪽서 리뷰어 손자까지 정리하는가? (codex/agy 내부 spawn 구조 의존)
2. node 부재가 정말 0에 수렴하나 — 리뷰어 CLI가 향후 비-node(rust 등)로 가면 전제 붕괴?
3. 프롬프트 stdin 전달 시 codex/claude/agy 각 CLI가 stdin 프롬프트를 지원하나(argv만 받으면 N11 무효)?
4. 쉘+`.mjs` 이중 보유 = 유지보수 2배 — 폴백 보존이 슬림 위반인가, 안전마진인가?
5. `.mjs` 단위테스트를 어디서 돌리나 — 레포에 node 테스트 러너(무의존 `node --test`) 도입이 슬림 허용 범위인가?
